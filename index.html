Cartas Pokemon (GSFP DAW/DAM): Guía Docente

Descripción

Se trata de un proyecto de desarrollo de software por grupos utilizando metodologías ágiles (EduScrum). El objetivo es desarrollar una aplicación web que consume la PokéAPI para generar 6 cartas Pokémon aleatorias. Cada carta aplica estilos CSS según el tipo del Pokémon (fuego → degradado rojo, agua → azul, planta → verde…) y existe una probabilidad baja (~1/256, guiño a la mecánica original) de que el Pokémon salga shiny, lo que activa un estilo visual diferenciado.

La actividad empieza por la fase de análisis: antes de escribir una línea de código, el alumnado modela el dominio mediante un diagrama de clases UML (clase Pokemon con sus atributos y métodos, clase controladora de la webapp, relaciones de composición/agregación con Type, Stat, Card).

Después se pasa a desarrollar el código sobre un repositorio base proporcionado por el docente (estructura HTML + CSS + JS lista, con el fetch de la API por implementar). Aplicarán contenidos de marcas (HTML semántico), presentación (CSS, variables, grid/flex), programación (clases, métodos, async/await, manejo de errores) y consumo de API REST con JSON.

Variante Java: quien prefiera puede implementar la lógica de modelado y mapeo de datos en Java como ejercicio paralelo (consumo de la API con HttpClient, parseo con Jackson/Gson, generación de un informe), reforzando POO del módulo 0485.

Agrupamientos y organización espacial

Aplicando los principios DUA, se combinan dos agrupamientos:

  1. Equipos eduScrum de 3-4 para los sprints de desarrollo: mesas juntas formando islas, ordenadores en disposición que permita ver la pantalla del compañero (programación en parejas rotativa).
  2. Gran grupo en los Sprint Reviews y retrospectivas: disposición en U para que todos se vean.

Los equipos se forman buscando heterogeneidad (mezcla de niveles de partida, perfiles más visuales con más analíticos). Se asigna un espacio en la pared para el tablero Kanban físico de cada equipo (post-its), complementario al digital.

Temporalización

8 sesiones de 1h y 50 minutos, organizadas con ABP + eduScrum (sprints cortos, Definition of Done, retrospectiva).


Sesión Fase Contenido Entregable
1 Kick-off Presentación del reto, exploración de la PokéAPI, formación de equipos (3-4), roles eduScrum (PO de equipo, SM, devs) Backlog inicial en tablero
2 Análisis Diagrama de clases del dominio (Pokemon, Type, Stat) Borrador UML
3 Análisis  Diagrama de la webapp (controlador, vista, servicio API) + planificación Sprint 1 UML completo + Sprint backlog
4 Sprint 1 Maquetado HTML + CSS base de la carta, sin datos reales Plantilla de carta estática
5 Sprint 2 Consumo de la PokéAPI, modelo Pokemon en JS, render de 6 cartas App funcional v1 + review
6 Sprint 3 Estilos por tipo (CSS variables / data-attributes), aleatoriedad Cartas con estilos por tipo
7 Sprint 4 Lógica shiny, refinamiento, testing manual App funcional v2
8 Cierre Grabación del vídeo demo, entrega, retrospectiva final Entrega completa

Productos a elaborar

  1. Infografía interactiva sobre la fase de análisis: explica el diagrama de clases, las decisiones de modelado (¿por qué Type es una clase aparte? ¿composición o agregación?) y el flujo de datos. Formato sugerido: Genially o Canva.
  2. Webapp en repositorio de GitHub (fork del repo de partida): código + README.md con instrucciones, capturas y reparto de tareas.
  3. Vídeo demo de 2-3 min mostrando la app en funcionamiento: 6 cartas generadas, refresh, aparición de un shiny (forzar la probabilidad para la grabación), explicación oral breve. Subido a Drive o YouTube no listado.

Herramientas Tecnológicas

Herramientas:

  • VSCode (editor) con extensiones recomendadas: Live Server, Prettier, GitLens
  • Navegador con DevTools (Network, Console)
  • GitHub: cuenta personal, fork del repo de partida, commits significativos
  • PlantUML / Mermaid / draw.io / StarUML para el diagrama de clases
  • Genially / Canva para la infografía interactiva
  • OBS / Loom / QuickTime para grabar el vídeo
  • NotebookLM como apoyo de estudio: cargar la documentación de la PokéAPI y los apuntes para generar resúmenes y podcasts

Recursos:

  • Repositorio de partida
  • Apuntes de los tres módulos (Aula Virtual)
  • Documentación oficial de la PokéAPI
  • MDN Web Docs para HTML, CSS y JS (preferible a W3Schools como referencia primaria)
  • W3Schools como apoyo rápido de sintaxis
  • Canales YouTube: Midudev, Carlos Azaustre, Hola Mundo...
  • CLAUDE.md en el repo: contexto del proyecto, convenciones de código, prompts útiles
